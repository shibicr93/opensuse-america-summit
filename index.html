Stage 1: ArgoCD out of the box. Git is your source of truth. ArgoCD detects drift between Git and the cluster, then syncs it back. Simple. Dangerous because the system doesn't know WHY something changed - only THAT it changed.

The Problem

  • Production Outages
    Rigid sync rules cause crashes under unexpected load or legitimate scaling.
  • Security & Compliance
    Unauthorized changes bypass the entire GitOps model - the system has no way to detect intent.
  • No Room for Nuance
    Every drift is treated equally - no way to distinguish between legitimate changes and actual problems.
GitOps works in theory: live state differs from Git, sync it back. Reality is messier. We get outages because rigid sync rules crash systems under load. We get security holes when unauthorized changes bypass the model. We lose nuance because every drift is treated equally. GitOps today is binary. Real operations need context.

  • Rego Policies - platform teams define drift rules without code changes
  • Sync Windows - OPA output schedules deferred syncs to maintenance windows
  • Policies can't learn from patterns - still a static rulebook
We brought in OPA to kill the hardcoding. Rego policies let platform teams define rules without touching code. The constraint: policies are static. They can't learn from patterns or handle ambiguity.

The Protocols: MCP & A2A

Model Context Protocol

AI to Tools - A standard interface for LLMs to call external tools

  • Exposes tools as typed JSON-RPC functions
  • Agent calls query_prometheus("rate(http_requests[5m])")
  • Server returns structured data - not HTML, not dashboards
  • Same MCP server works with any LLM provider

Agent-to-Agent (A2A)

Agent to Agent - Agents discover and delegate to each other

  • Drift Agent delegates to Security Agent for RBAC analysis
  • Security Agent delegates to Compliance Agent for audit
  • Each agent has its own MCP tools and specialization
  • Agent Cards advertise capabilities for discovery
Two protocols make Stage 3 work. MCP - Model Context Protocol - is how the AI agent calls external tools. A2A - Agent to Agent - is how agents discover and delegate to each other.

Guardrails: Trusting AI in Production

Safety Boundaries

  • AI can ALLOW or DEFER - never DELETE
  • Confidence threshold: below 80% → ask human
  • OPA policies act as hard guardrails
    "Never auto-approve image tag changes"
  • Shadow mode: run alongside existing sync for 30 days

Observability

  • Every decision logged with full reasoning chain
  • Grafana dashboard: drift decisions / hour
  • Alert on AI ≠ OPA disagreements
  • Weekly accuracy review: was the AI right?
Shadow Mode observe only
validate ───▶
Advisory Mode suggest + confirm
promote ───▶
Autonomous Mode act within bounds
Should you let AI make production decisions? Not without guardrails. The agent can only ALLOW or DEFER - never delete. Below 80% confidence, it asks Slack.

Key Takeaways

  1. Not all drift is equal - scaling a deployment isn't the same as deleting a secret. Your tooling should understand the difference and respond accordingly.
  2. ArgoCD already has the building blocks - Custom Health Checks, Resource Hooks, ApplicationSets, and Sync Windows. You don't need to replace it - you need to extend it.
  3. Guardrails are non-negotiable - start with OPA policies. Only bring in AI for the ambiguous cases. And always: shadow mode first, human-in-the-loop, hard boundaries.
Three takeaways. First: not all drift is equal. Second: ArgoCD gives you the blocks. Third: guardrails are non-negotiable. OPA handles 90% of cases. Use AI for the ambiguous 10%.
